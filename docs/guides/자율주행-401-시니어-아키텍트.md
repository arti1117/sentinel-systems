# 🧠 자율주행 401: 시스템 아키텍트를 위한 자원 격리 및 결정론적 설계

이 과정은 단순한 소프트웨어 개발을 넘어, **운영 체제 커널(Linux Kernel)**과 **미들웨어(ROS 2 / DDS)** 단의 한계를 고민하는 상위 아키텍트를 위한 문서입니다.

소프트웨어가 아무리 잘 짜여져 있더라도 자원이 펌웨어 레벨에서 격리(Isolation)되지 않는 한, 프로덕션의 극단적인 시나리오 속에서 오작동(비결정론적 멈춤)은 예정된 수순입니다.

---

## 1. 단일 스레딩(Single-Threaded)과 멀티 스레딩의 딜레마

ROS 2 아키텍처의 심장인 **Executor(실행기)** 디자인은 C++ 어플리케이션 런타임의 생사를 가릅니다.

### Single-Threaded Executor의 숨겨진 치명타
대부분의 ROS 2 샘플 코드는 단일 스레드 실행기를 사용합니다.
- **문제점**: 이미지 콜백(Perception) 하나가 처리에 100ms를 점유하고 있다면, 그 스레드가 풀려날 때까지 브레이크를 밟아달라는 1ms짜리 비상 제어 콜백 메시지는 큐(Queue)에서 영원히 블로킹(Blocking)됩니다. 한 모듈의 지연이 전체 시스템의 안전 계통을 완전히 마비시켜 차량 직진 사고를 냅니다.

### Callback Group Isolation (멀티 스레딩의 제어)
이러한 Head-of-Line 패널티를 막기 위해 아키텍트는 **서로 상호 배타적인(Mutually Exclusive) 콜백 그룹**을 설계해야 합니다.
- 무거운 인지 작업 콜백과 가벼운 제어/비상 정지 루프를 절대 같은 스레드 풀(Thread Pool)에 묶지 않습니다. 멀티스레드 Executor를 사용하되 데드락(Deadlock)과 Race Condition이 발생하지 않도록 메모리 접근 권한을 락-프리(Lock-Free) 링 버퍼 형식으로 엄격히 재단하는 것이 시스템의 최우선 기본기입니다.

## 2. Linux OS의 스케줄러 간섭 방어 (CPU/Memory Pinning)

기본 Linux (우분투) 시스템의 OS 커널 관리자(CFS 스케줄러)는 자율주행 프로세스가 실시간 성능을 발휘하는 환경에 적합하지 않습니다.

### CPU 친화성 (Affinity) 및 Pinning
- OS가 자율주행 핵심 노드(MPC 제어 등)를 1번 코어에서 3번 코어로 스와핑(Swapping)할 때 캐시 메모리가 날아가며(Cache Miss) 레이턴시 통제가 터집니다.
- **해결책**: 백그라운드 I/O 작업(네트워크 통신 등)을 특정 코어(예: 0번 1번)에 가두고, 핵심 자율주행 프로세스를 격리된 전용 코어(예: 2, 3, 4번)에 고정(CPU Pinning)시킵니다.

### Memory Page Fault 차단
- 메모리를 아끼기 위해 OS가 램(RAM) 영역을 디스크(SWAP)로 내리게 되면 재요청 시 엄청난 지연 타임 아웃이 발생합니다.
- 아키텍트는 `mlockall()` 커널 시스템 콜을 이용해 자율주행에 쓰이는 모든 프로세스 메모리가 영구적으로 물리 램(RAM)에 박혀 내려가지 않도록(Page Fault 회피) 하드 리얼타임 구조를 강제 수립해야 합니다.

## 3. 차세대 미들웨어 오버헤드 통제: DDS Security와 Trade-off

최근 자율주행의 B2B 법적 요구사항이 거세지면서, 차내 네트워크를 암호화하기 위해 **DDS Security(PKI 기반)**가 반강제적으로 들어오고 있습니다.
여기에서 심각한 마찰(Friction)이 발생합니다.

- **암호화 연산 병목**: LiDAR 포인트 클라우드(1초당 천만 개의 Float 값)에 일일이 타원 곡선 보안(S/MIME 서명 및 암호화)을 적용하면 CPU 점유율이 폭등하고 레이턴시가 파괴적 수준(10배 이상 지연)으로 증가합니다.
- **아키텍처적 Trade-off 방침 분리**: 이 지점에서 모든 내부 데이터 통신에 맹목적 암호화를 수행하지 말고 **격리(Enclave)** 접근을 취합니다. 대용량/센서 데이터 통신은 Zero-Copy/Non-secure 메모리 포인터 교환으로 오버헤드를 제로화하고, 반면 자율주행의 핵심 플래닝(명령 하달) 메시지 큐 등 극소량의 통제 도메인에 한증하여 접근 제어 방벽(Access Control Payload Security)을 적용하는 이원화 정책 설계가 반드시 선행되어야 합니다.

## 4. Sentinel Systems의 클라우드 브리지 관점에서의 연계

차량 안에서 아무리 하드 리얼타임 격리와 자원 고정(Pinning)이 잘 이루어지더라도, 결국 관제 시스템과 엮이는 아웃바운드 엣지에서 파멸할 수 있습니다. 

- 차량이 아무리 방어를 굳건히 해도, 클라우드와 통신하는 데몬이 차량 내부의 CPU/네트워크를 포식하게 둔다면 배보다 배꼽이 커지게 됩니다. 
- 이 관점에서 향후 Sentinel Systems는 차량 메인 두뇌(ECU)가 아닌 완전히 분리된 별도의 네트워크/보안 프록시 칩(TCU) 단에서 텔레메트리 관제 데이터를 집계 및 탈출시키는 "비공유격리 브리지(Non-Shared Isolation Bridge)" 설계안 등 하드웨어 독립적인 아키텍처 지원 모델을 구상해야 합니다. 그래야만 Sentinel Agent의 오작동이 자율주행 메인 루틴의 레이턴시에 조금도 영향을 주지 않는 궁극적 무결성 원칙을 검증해 낼 수 있기 때문입니다.
