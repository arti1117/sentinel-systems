# 🧠 자율주행 401: 시스템 아키텍트를 위한 자원 격리와 결정론적 최적화 설계

[👈 이전 과정 301](./자율주행-301-엔지니어-딥다이브.md) | [🏠 인덱스로 돌아가기](./README.md) | [다음 과정 501 👉](./자율주행-501-최고전문가-CTO.md)

일반적인 딥러닝 애플리케이션 개발 방법론과 차원을 달리하는 본 과정은 **분산 운영 체제(Linux Kernel)** 및 **실시간 통신 미들웨어(ROS 2)** 계층을 통합 조율해야 하는 수석 아키텍트를 위해 쓰여졌습니다.

인공지능 코드가 아무리 우아해도 메모리와 하드웨어 자원이 커널 레벨에서 분리(Isolation)되지 않았다면, 그 시스템의 오작동 확률과 교착 시간은 오직 신만이 결정하게 됩니다. 목표는 확률론적 체계를 지우고 결정론적 반응(Deterministic Reaction) 시간의 완전 장악입니다.

---

## 1. 운영 체제의 선점과 극복 수단 (CPU/Memory Pinning)

범용 우분투(Ubuntu Linux) 내부의 중앙 스케줄러(CFS)는 자율주행 차량이 직면한 밀리초 단위의 치명성(Criticality) 따위는 전혀 안중에도 없습니다.

### 하드 코어 고정 전략 (CPU Affinity Configuration)
- **배경 지식 위험**: 운영 체제 스케줄러가 특정 자율 판단 알고리즘 코드를 CPU 1번에서 CPU 3번 스레드로 강제 이주(Swapping) 시킬 경우, 코어 내부 L1/L2 메모리 버퍼가 삭제되며 대량의 연산 지연(Cache Miss)이 발생합니다.
- **설계 해법 해제**: 핵심 연산 제어 노드는 철저히 리눅스 커널 프로세싱에서 분리해 전용 고립 코어 구조(Isolated Cores, 예: Core 2,3,4)에 묶어버립니다 (CPU Pinning). 대신 와이파이 네트워크 통신 등 가방끈이 긴 외부 작업은 Core 0에 수용해 버려 두 도메인이 상호 침범하는 것을 커널 아키텍처 단위로 통제합니다.

### 메모리 페이지 폴트(Page Fault)의 사형 선고 피하기
- 메모리 공간 압박 때문에 OS가 활성 메모리(RAM)를 느려터진 저장소(Swap Disk)로 유출했다가 재요구하는 경우, 그 시간 지연 속에서 차량 충돌이 확정됩니다.
- 아키텍트는 리눅스 시스템 호출인 `mlockall()`명령어를 스택에 설계 삽입해, 자율주행 구동 프로세스의 물리 공간이 평생 동안 메인 램 공간에서 뜯겨나가지 못하도록 강제 무결성을 세팅합니다 (Hard Real-time Page Fault Avoidance).

## 2. 단일 스레드 구조의 구조적 치명타와 Callback Group 격리 철학 

ROS 2(로봇 운영 체제 2) 내장형 스케줄러를 설계할 때 가장 심각한 함정은 노드들 간의 우선 권한 처리를 관장하는 실행 지시자(Executor)의 선택 패턴입니다.

### 병행 처리의 함정과 교착 상태 (Deadlock)
많은 개발자가 쉽게 생각하여 기본 제공 단일 스레드 실행자(Single-Threaded Executor)를 적용합니다. 이 상황에서는 거대한 사진 한 장을 분석하느라 100ms를 점유하는 인지 호출기계 뒤쪽 대기열(Queue)에 있는 "초긴급 1ms 브레이크 밟아라" 명령이 그 연산이 해제될 때까지 영원히 봉인되는 블로킹(Blocking) 비상 정지 불능 패널티 구조를 양산합니다.

### 상호 배타적 콜백 격리망 아키텍처 (Mutually Exclusive Callback Group)
따라서 아키텍트는 **다중 스레드 구동자 (Multi-Threaded Executor)** 체제로 프로세스를 전환해야 하나 이로 인한 또 다른 메모리 무단 공유 경합 조건(Race Condition)을 원천 방어해야 합니다.

```text
             [ 운영 체제 (OS) ]
               │            │
  스레드 0번 할당 │            │ 다수 스레드(Thread Pool) 할당
               │            │
               ▼            ▼
┌────────────────────────────────────────────────────────────┐
│         ROS 2 Multi-Threaded Executor (실행 블록 분할)       │
│                                                            │
│  [ Group 1: MutuallyExclusive Callback Group ]             │
│  ┌──────────────────────────────────────────────────────┐  │
│  │ ( 저지연 고가용 긴급 탈동 제어 : 1000Hz 잦은 호출 )      │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                            │
│  [ Group 2: Reentrant Callback Group ]                     │
│  ┌──────────────────────────────────────────────────────┐  │
│  │ ( 레이저 포인트 전처리 변환 프로세스 )                   │  │
│  │ ( 이미지 연산 프로세스 배열 )                            │  │
│  └──────────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────────┘
```

- 도표와 같이 **절대 상호 간에 양보해서는 안 될 지연 보장 노드 그룹(Group 1)**과 **대용량 파이프라인 처리가 우선인 재배치 수용 그룹(Group 2)**의 풀장(Pool)을 소프트웨어 설계 수준에서 하드 코딩 분할합니다. 서로 접근할 수 없게 데이터 교환 구조는 비잠금 링 버퍼(Lock-Free Ring Buffer)로 단절시켜 메모리 충돌 락킹(Deadlock)이 본래 일어나지 않게 구조적 통로를 막아야만 합니다.

## 3. 프로덕션 아웃바운드 관점 : 엣지 텔레메트리와 클라우드 백본망 (Sentinel Architecture)

차내 두뇌가 결정론적 완벽성을 달성하더라도 관제탑 기지로 데이터를 송출하는 과정의 외부 통신이 자동차 내부 네트워크 자원을 갉아먹는다면(통신 로드 폭증에 의한 패킷 헤드 블로킹) 무의미해질 것입니다.

센티넬 시스템즈(Sentinel Systems) 통합 백본의 확장 방향은 차량 핵심 CPU/메모리 위에서 외부 통신기를 작동시키는 단순 데몬 솔루션에서 벗어나야 합니다. 
대신 차량 내 독립적 통신 칩 단자(TCU, Telecommunication Control Unit)에서 비공유 보안 포켓(Non-Shared Security Proxy)을 열어 두고 차내 내부 통신 로컬 영역에는 그저 엿듣기(Packet Sniffing) 형태로 정보를 강탈하는 백도어(Backdoor) 인프라로 연결되는 청사진 설계. 이를 수립해야만 핵심 제어 노드에 0.001초의 관제 데이터 송신 부담도 내리지 않는 극한의 결정론적 분리(Deterministic Segregation) 관제 솔루션을 팔 수 있게 됩니다.
