# 자율주행 시스템 종합 분석

> 워크스페이스 내 3개 프로젝트 그룹(AutowareFoundation · CarlaSimulator · F1Tenth)과 Sentinel Systems 비즈니스 레이어를 통합 분석합니다.

---

## 1. 전체 구조 — 세 개의 프로젝트가 하나의 시스템인 이유

```text
┌──────────────────────────────── 가상 세계 ────────────────────────────────┐
│                       [ CARLA (UE5 시뮬레이터) ]                          │
└───────┬────────────────────────────────────────────────────────┬──────────┘
        │ (ros-bridge / carla-autoware)                          │ (f1tenth_gym)
        ▼                                                        ▼
┌──────────────────────────────── 현실 세계 ────────────────────────────────┐
│ [ Autoware (실차 풀 스택) ] ──(동일 ROS 2 토픽)──▶ [ F1Tenth (1/10 스케일) ] │
└───────┬────────────────────────────────────────────────────────┬──────────┘
        │                                                        │
        ▼                                                        ▼
┌───────────────────────────── 비즈니스 레이어 ─────────────────────────────┐
│                 [ Sentinel Systems (B2B 관제 인프라) ]                    │
└───────────────────────────────────────────────────────────────────────────┘
```

이 세 프로젝트는 독립적인 것이 아니라 **하나의 자율주행 개발 파이프라인**의 세 단계입니다:

| 단계 | 프로젝트 | 역할 | 리스크 |
|------|----------|------|--------|
| **1. 시뮬레이션** | CARLA | 가상 환경에서 무한 반복 테스트 | 현실과의 격차(Sim-to-Real Gap) |
| **2. 소규모 실증** | F1Tenth | 실제 물리 환경에서 저비용 검증 | 스케일 차이 (1/10 → 실차) |
| **3. 실차 배포** | Autoware | 프로덕션 레벨 자율주행 | 안전성, 규제, 운영 비용 |

**Sentinel Systems**는 이 세 단계 모두에 걸쳐 발생하는 **운영의 고통(Operational Pain Points)**을 B2B 관제 인프라로 제품화하는 비즈니스 레이어입니다.

---

## 2. 자율주행 파이프라인 — "사람의 운전"을 분해한 것

자율주행 시스템의 본질은 **인간 운전자가 무의식적으로 수행하는 인지-판단-조작 과정을 소프트웨어로 대체**하는 것입니다.

```text
[🔍 Sensing] ──▶ [📍 Localization] ──▶ [👁️ Perception] ──▶ [🗺️ Planning] ──▶ [🎮 Control] ──▶ [🚗 Vehicle]
센서 데이터 취득   내 위치는 어디?        주변에 뭐가 있지?       어디로 어떻게?        핸들/페달 조작      차량 구동
```

각 단계의 핵심을 프로젝트별로 비교합니다:

### 2.1 Sensing (센서 데이터 취득)

| 항목 | Autoware | CARLA | F1Tenth |
|------|----------|-------|---------|
| **LiDAR** | 실제 3D LiDAR (Velodyne 등) | GPU 레이캐스팅 시뮬레이션 | 2D Hokuyo LiDAR (1080빔, 270°) |
| **카메라** | RGB/Depth/Semantic | UE5 렌더링 + 17종 센서 레지스트리 | 미사용 (LiDAR Only) |
| **IMU/GNSS** | 실제 관성 센서 + GPS | 시뮬레이션 IMU/GNSS | 엔코더 기반 오도메트리 |
| **레이턴시** | 0-40ms | 10-50ms (Tick 기반) | <1ms (EDT 레이캐스트) |

> [!IMPORTANT]
> **Autoware의 센서 전략**: 현대차/모셔널과 같이 "비전 온리" 방식을 배제하고 **LiDAR+레이더+카메라 멀티모달 센서 퓨전**을 채택합니다. 이는 테슬라와의 핵심 차별점입니다.

### 2.2 Localization (위치 추정)

**"내가 지도상 어디에 있는가?"** — 자율주행의 가장 기초적이면서 가장 치명적인 문제.

#### Autoware: NDT + EKF 2단계 추정

```
[LiDAR 포인트클라우드]
    ↓
NDT Scan Matcher ────→ 정밀 자세 추정 (10Hz, 20-50ms)
  ├── 복셀 그리드 매칭 (해상도 1.0m)
  ├── OpenMP 병렬화
  ├── GNSS 정규화 (피처 부족 도로 대응)
  └── 4가지 공분산 추정 모드
    ↓
EKF Localizer ────→ 센서 융합 + 50Hz 출력
  ├── 6차원 상태: [x, y, yaw, yaw_bias, vx, wz]
  ├── 시간 지연 보상 (0-500ms 센서 딜레이)
  ├── 마할라노비스 거리 게이팅 (이상치 거부)
  └── 요 바이어스 자동 보정
    ↓
TF 브로드캐스트: map → base_link
```

> [!NOTE]
> **NDT(Normal Distributions Transform)**: 사전 구축된 3D 포인트클라우드 맵과 실시간 LiDAR 스캔을 정합(Registration)하여 차량의 6-DOF 자세를 추정합니다. GPS가 불안정한 도심 환경에서 cm 수준 정확도를 제공하는 핵심 기술입니다.

#### F1Tenth: MCL (Monte Carlo Localization)

```
[2D LiDAR 스캔]
    ↓
Particle Filter ────→ 위치 추정
  ├── 수천 개의 "추측 위치" 파티클 분포
  ├── range_libc로 고속 레이캐스팅 (C++/CUDA)
  └── 관측 우도 기반 리샘플링
    ↓
오도메트리: VESC 엔코더 → 자전거 모델 적분
```

**핵심 차이**: Autoware는 3D 맵 + NDT로 cm 수준 정밀도, F1Tenth는 2D 맵 + 파티클 필터로 저비용 실시간 추정.

### 2.3 Perception (주변 환경 인식)

#### Autoware: CenterPoint 3D 객체 검출

```
PointCloud2 입력
    ↓
[복셀화 + 필라 인코딩] ← TensorRT GPU 가속
    ↓
[백본 + DetectionHead] ← ONNX 모델
    ↓
[후처리: NMS + 시간적 밀도화]
    ↓
DetectedObjects: [CAR, TRUCK, BUS, BICYCLE, PEDESTRIAN]
```

- **감지 범위**: 102.4m × 102.4m (XY), 8m (Z)
- **복셀 해상도**: 0.2m × 0.2m × 8.0m
- **GPU 메모리**: 2-4 GB
- **레이턴시**: 50-100ms @ 10Hz

#### CARLA: Ground Truth 제공

CARLA는 인식 알고리즘을 개발하는 것이 아니라, **학습 데이터와 테스트 시나리오를 제공**합니다:
- 시맨틱 세그멘테이션 카메라 → CityScapes 라벨 (Ground Truth)
- DVS(이벤트) 카메라 → 스파이크 스트림
- 17종 센서 시뮬레이션 → 학습용 데이터 대량 생성

#### F1Tenth: LiDAR Only

```
2D LiDAR 스캔 (1080빔, 30m)
    ↓
EDT(유클리드 거리 변환) 기반 레이 트레이싱
    ↓
장애물 거리 배열 → 직접 Planning에 전달
```

별도의 딥러닝 인식 파이프라인 없이, LiDAR 거리값을 직접 제어 알고리즘에 입력합니다. **단순하지만 10ms 이내 End-to-End 레이턴시**의 비결입니다.

### 2.4 Planning (경로 계획)

#### Autoware: 계층적 플러그인 아키텍처

```text
┌─────────────────────────────────────────────────────────┐
│ [ Mission Planner ] : 목적지까지 글로벌 경로 탐색       │
└────────────────────────────┬────────────────────────────┘
                             ▼
┌─────────────────────────────────────────────────────────┐
│ [ Behavior Planner ] : 상황별 행동 결정                 │
│  ├─ Lane Following (차선 유지)                          │
│  ├─ Lane Change (차선 변경)                             │
│  ├─ Obstacle Avoidance (장애물 회피)                    │
│  └─ Goal/Start Planner (출발/도착 계획)                 │
└────────────────────────────┬────────────────────────────┘
                             ▼
┌─────────────────────────────────────────────────────────┐
│ [ Trajectory Planner ] : 실시간 궤적 생성 및 보간       │
└─────────────────────────────────────────────────────────┘
```

- **Scene Module 플러그인**: Lane Following, Static Obstacle Avoidance, Lane Change, Goal Planner, Start Planner 등
- **궤적 보간**: AkimaSpline(5점), CubicSpline(4점), Linear(2점)
- **Lanelet2 맵**: 차선, 교차로, 신호등, 규제 정보 포함 정밀 지도

#### CARLA: OpenSCENARIO Behavior Tree

```
OSC2 시나리오 (.osc2)
    ↓
ANTLR4 파서 → AST → Symbol Manager
    ↓
py_trees Behavior Tree 실행
    ↓
Atomic Behaviors → CARLA API 호출
```

- **Traffic Manager**: 4단계 파이프라인 (Localization → Collision → TrafficLight → MotionPlan)
- **PID 제어**: 도심 Kp=1.0, 고속도로 Kp=0.8
- 곡률 기반 감속: 3점 원 반경 계산

#### F1Tenth: 경량 알고리즘 라이브러리

`f1tenth_planning` 패키지에 포함된 알고리즘:

| 카테고리 | 알고리즘 |
|----------|----------|
| **Planning** | Lattice Planner, Graph-based, FGM, Wall Follower |
| **Control** | MPC, Pure Pursuit, Stanley, LQR, PID |

별도의 고정밀 지도 없이, 레이스라인 CSV 데이터(`s_m; x_m; y_m; psi_rad; kappa_radpm; vx_mps; ax_mps2`)를 기반으로 동작합니다.

### 2.5 Control (차량 제어)

#### Autoware: 선형 MPC + OSQP

```
기준 궤적 → [경로 필터링] → [MPC QP 최적화] → [Butterworth LPF] → 조향 출력

비용함수: J = Σ[w_lat·e_lat² + w_head·e_head² + w_steer·δ² + w_jerk·j²]
```

- **차량 모델**: Kinematics(자전거 모델 + 1차 조향 지연) 또는 Dynamics(슬립각 포함)
- **QP 솔버**: `unconstraint_fast`(빠름) 또는 `OSQP`(제약조건 포함)
- **예측 수평선**: 50-100 스텝, dt=0.05-0.1s
- **레이턴시**: 10-20ms @ 50Hz

#### F1Tenth: 다양한 제어기 선택

| 제어기 | 특징 | 용도 |
|--------|------|------|
| **Pure Pursuit** | 전방 목표점 추종, 단순/빠름 | 기본 주행 |
| **Stanley** | 횡방향 오차 + 헤딩 오차 | 정밀 추종 |
| **MPC** | 미래 예측 최적화 | 고속 레이싱 |
| **LQR** | 선형 최적 레귤레이터 | 안정화 |
| **PID** | 비례-적분-미분 | 속도 제어 |

## 3. 시뮬레이션 생태계 (CARLA) — 가상 세계의 구조

### 3.1 클라이언트-서버 아키텍처

```
[Python/C++ Client] ←── MsgPack RPC ──→ [UE5 Server]
       │                                      │
  LibCarla (C++)                        Chaos Physics
  PythonAPI (SWIG)                      Sensor Actors (17종)
                                        Traffic Manager
                                        OpenDRIVE 맵
```

- **동기화 모드**: `world.tick()` → 모든 물리/센서 업데이트 → 결과 반환
- **비동기 모드**: 서버가 독립적으로 물리 연산, 클라이언트는 비동기 수신
- **차량 물리**: 엔진 토크 → 변속기 → 디퍼렌셜 → 휠 포스 (UE5 Chaos Physics)

### 3.2 연결 생태계

```text
[ CARLA (코어 시뮬레이터 UE5) ]
  ├──▶ [ scenario_runner ] (시나리오 실행 엔진)
  ├──▶ [ ros-bridge ] (ROS 1/2 연결 다리)
  ├──▶ [ leaderboard ] (성능 평가 벤치마크)
  ├──▶ [ map ] (OpenDRIVE 파싱)
  ├──▶ [ Scenic ] (확률적 시나리오 생성기)
  ├──▶ [ sumo ] (대규모 교통 흐름 연동)
  ├──▶ [ RobotecGPULidar ] (GPU 연산 가속 LiDAR)
  ├──▶ [ carla-digitaltwins ] (OSM → 가상맵 변환)
  └──▶ [ imitation-learning / reinforcement-learning ] (모방 및 강화학습 프레임워크)
```

> [!WARNING]
> **버전 호환성**: CARLA 생태계의 최대 난제. DReyeVR(0.9.13), carla-autoware(0.9.10.1), rllib(0.9.11)이 각각 다른 CARLA 버전에 고정되어 있어 동시 사용이 불가능합니다. 사용할 도구 조합을 먼저 정하고, 거기에 맞는 CARLA 버전을 선택해야 합니다.

### 3.3 학습 파이프라인

CARLA 위에서 두 가지 학습 방식이 구현되어 있습니다:

| 방식 | 저장소 | 아키텍처 | 핵심 |
|------|--------|----------|------|
| **모방 학습 (CIL)** | `imitation-learning` | CNN 인코더 + 조건부 브랜치 분기 | 전문가 시연 데이터 → 조향/가속/브레이크 회귀 |
| **강화 학습 (A3C)** | `reinforcement-learning` | NatureDQN + Actor-Critic | 보상 기반 비동기 멀티프로세스 학습 |

## 4. 실차 시스템 (F1Tenth) — 가장 작은 자율주행차

### 4.1 하드웨어-소프트웨어 스택

```
joy → joy_teleop ──┐
                    ├→ ackermann_mux → ackermann_to_vesc → vesc_driver → VESC ESC
autonomous_node ───┘                                                      ↓ (serial)
                                                                      DC 모터

urg_node (Hokuyo LiDAR) → /scan (LaserScan, 1080빔)
vesc_to_odom → /odom (Odometry) + TF (odom → base_link)
Static TF: base_link ↔ laser [x=0.27m, z=0.11m]
```

### 4.2 물리 모델 — 왜 중요한가

F1Tenth의 물리 시뮬레이션은 **두 가지 차량 모델**을 제공합니다:

| 모델 | 자유도 | 상태 벡터 | 적용 조건 |
|------|--------|-----------|-----------|
| **Kinematic (KS)** | 5-DOF | `[x, y, δ, v, ψ]` | 저속 (`v < 0.5 m/s`) |
| **Dynamic (ST)** | 7-DOF | `[x, y, δ, v, ψ, ψ̇, β]` | 고속 (타이어 슬립 발생 시) |

> [!TIP]
> **Sim-to-Real 핵심**: F1Tenth의 Gym 시뮬레이터와 실차가 **동일한 ROS 토픽**(`/scan`, `/drive`, `/odom`)을 사용하므로, 시뮬레이션에서 개발한 코드가 수정 없이 실차에서 동작합니다. 이것이 F1Tenth 플랫폼의 가장 큰 가치입니다.

### 4.3 안전 메커니즘

- **데드맨 스위치**: 조이스틱 LB(텔레옵) / RB(자율주행) 버튼 — 놓으면 즉시 정지
- **Throttle Interpolator**: 사다리꼴 보간으로 순간 토크 스파이크 방지 (75Hz)
- **서보 포화 클리핑**: 조향각, 속도의 물리적 한계 강제 적용
- **iTTC 충돌 예측**: Time-to-Collision < 0.005초 시 충돌 판정

## 5. 성능 비교 — 숫자로 보는 세 시스템

### 5.1 End-to-End 레이턴시

| 컴포넌트 | Autoware | CARLA | F1Tenth |
|----------|---------|-------|---------|
| 인지/센서 | 50-100ms | 10-50ms | <1ms |
| 위치 추정 | 20-50ms (NDT) | — | <1ms |
| 제어 | 10-20ms (MPC) | <1ms (PID) | <1ms (PID) |
| **E2E 총합** | **200-300ms** | **10-50ms** | **~10ms** |

> [!IMPORTANT]
> Autoware의 200-300ms E2E 레이턴시는 실차의 안전 마진 확보를 고려하면 **60km/h 기준 약 3.3-5m의 반응 거리**를 의미합니다. F1Tenth의 10ms는 1/10 스케일에서 약 0.08m로, 스케일 비율을 고려하면 유사한 수준입니다.

### 5.2 핵심 알고리즘 비교

| 도메인 | Autoware | CARLA | F1Tenth |
|--------|----------|-------|---------|

## 6. Sentinel Systems의 포지셔닝 — 어디에 가치를 만드는가

### 6.1 이 세 프로젝트에서 발견되는 "운영의 고통"

위 기술 분석에서 드러난 운영 마찰 지점들:

| 고통 | 구체적 증상 | Sentinel 기회 |
|------|------------|---------------|
| **버전 호환성 카오스** | CARLA 0.9.10 ~ 0.10.0까지 5개 이상 버전이 혼재 | 호환성 관리/배포 자동화 |
| **센서 캘리브레이션** | NDT 파라미터 12개+, EKF 파라미터 10개+ 수동 튜닝 | Zero-Config 자동 세팅 |
| **장애 진단 불투명** | NDT 수렴 실패, EKF 게이팅 거부 시 원인 파악 어려움 | 비개발자용 진단 대시보드 |
| **시뮬-실차 격차** | Gym에서 잘 되던 코드가 실차에서 다르게 동작 | Sim-to-Real 격차 정량화 도구 |
| **빌드 의존성 지옥** | range_libc → particle_filter 순서, 117개 저장소 빌드 | 원클릭 환경 구축 |

### 6.2 비즈니스 전략과의 정렬

[Business Start](file:///home/jy/Documents/JY/sentinel-systems/docs/strategy/Business%20Start.md) 문서의 필터에 대입:

```
✅ Structural Legacy    — 관제 인프라는 재사용 가능한 구조적 자산
✅ Scalability          — 차량 대수 증가에 비례하여 확장
✅ Human Intervenability — 관제 = 인간 개입 지점 자체가 제품
✅ Transferability      — SaaS 형태로 문서화/이전 가능
✅ Business Connectivity — 차량 대당 월정액 구독 수익 모델
```

5개 필터 **전부 충족** → 자원 투입 정당화.

### 6.3 시장 타이밍

[B2B 리서치](file:///home/jy/Documents/JY/sentinel-systems/docs/research/2026Q1-자율주행-로봇-B2B-리서치.md)에 따르면:

- **2026년 하반기**: 현대차/모셔널 서울 마포 레벨 4 로보택시 운영 시작
- **2027-2028년**: 자율주행 트럭 마일당 비용이 인간 운전 대비 역전
- **한국 자율주행 시장**: 3.6조원(2025) → 26.2조원(2035), CAGR ~22%

> [!CAUTION]
> **타이밍 리스크**: 관제 인프라 수요는 실제 플릿 배포 규모에 비례합니다. 현대차/모셔널의 2026년 하반기 로보택시 운영이 **예정대로 진행되는지**가 Sentinel Systems의 시장 진입 타이밍을 결정하는 핵심 변수입니다.

## 7. 핵심 정리

| 질문 | 답 |
|------|-----|
| 자율주행이란? | 인간의 인지-판단-조작을 Sensing→Localization→Perception→Planning→Control로 분해하여 소프트웨어로 대체하는 것 |
| 왜 3개 프로젝트인가? | 시뮬레이션(CARLA) → 소규모 실증(F1Tenth) → 실차 배포(Autoware)의 개발 파이프라인 |
| 가장 어려운 부분은? | 알고리즘이 아니라 **운영**(배포, 튜닝, 장애 진단, 버전 관리) |
| Sentinel Systems는? | 이 운영 고통을 B2B SaaS로 제품화하는 관제 인프라 사업 |
| 시장 규모는? | 한국 자율주행 시장만 2035년 26.2조원, 관제/RaaS CAGR 21%+ |
